# 远端脚本注入

这里直接打开：https://notes.edmund.xin/demo/tiny.user.js 【因为写了开头的// ==UserScript==】触发脚本安装，match触发初始化会先require远端包，然后会先初始化远端包【实际上是类似加载到本地相当于就是本地文件】，之后再加载脚本代码

这个 demo 的链路是：

```txt
tiny.user.js
  -> 通过 @require 请求远端 tiny-widget.js
  -> tiny-widget.js 先执行，声明 TinyFloatDemo()
  -> tiny.user.js 正文再执行，调用 TinyFloatDemo()
  -> TinyFloatDemo() 操作 DOM，往当前网页 append 一个悬浮方块
```

## @require 做了什么

`tiny.user.js` 里有这一行：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
```

脚本管理器看到 `@require` 后，会在运行 `.user.js` 正文之前，先下载并执行这个远端 JS 文件。

执行顺序是：

```txt
1. 用户安装 tiny.user.js
2. 脚本管理器读取 userscript metadata
3. 发现 @require
4. 请求 https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
5. 下载并执行 tiny-widget.js
6. tiny-widget.js 声明全局函数 TinyFloatDemo()
7. 再执行 tiny.user.js 正文
8. tiny.user.js 调用 TinyFloatDemo()
```

所以 `tiny.user.js` 可以直接写：

```js
TinyFloatDemo();
```

因为这个函数已经由 `tiny-widget.js` 提前声明好了。

## ?v=0.0.1 是什么

`?v=0.0.1` 不是后端版本匹配，也不是特殊协议。它只是 URL 的查询参数。

这个 URL：

```txt
https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
```

真正对应的静态文件路径是：

```txt
public/demo/tiny-widget.js
```

后面的：

```txt
?v=0.0.1
```

主要作用是破缓存。

对浏览器或脚本管理器来说，下面两个 URL 是不同的缓存键：

```txt
/demo/tiny-widget.js?v=0.0.1
/demo/tiny-widget.js?v=0.0.2
```

但对当前后端静态资源来说，它们仍然对应同一个文件：

```txt
/demo/tiny-widget.js
/demo/tiny-widget.js?v=0.0.1
/demo/tiny-widget.js?v=0.0.2
```

都会返回：

```txt
public/demo/tiny-widget.js
```

因为 URL 会被拆成两部分：

```txt
pathname = /demo/tiny-widget.js
search   = ?v=0.0.2
```

当前后端匹配静态资源时主要看 `pathname`，不会因为带了 `?v=0.0.2` 就请求不到。

所以当你把 `@require` 从：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
```

改成：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.2
```

脚本管理器会把它当成一个新的远端依赖地址，重新下载 `tiny-widget.js`。

## 后端有没有匹配版本号

当前没有。

当前 Cloudflare Worker 里的核心逻辑类似这样：

```ts
const url = new URL(request.url);

if (url.pathname === "/notes" || url.pathname.startsWith("/notes/")) {
  return handleNotesRequest(request, env, url);
}

if (env.ASSETS) {
  return env.ASSETS.fetch(request);
}
```

也就是说，只有 `/notes` 会进入 API 逻辑，其它路径都会交给 `public` 静态资源目录。

当请求是：

```txt
https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.2
```

解析出来是：

```ts
url.pathname // "/demo/tiny-widget.js"
url.search   // "?v=0.0.2"
```

它不会进入 `/notes` 分支，而是走：

```ts
env.ASSETS.fetch(request)
```

所以最终返回的还是：

```txt
public/demo/tiny-widget.js
```

除非你自己在 Worker 里写读取版本号的逻辑：

```js
const version = new URL(request.url).searchParams.get("v");
```

例如：

```ts
const url = new URL(request.url);
const version = url.searchParams.get("v");

if (url.pathname === "/demo/tiny-widget.js") {
  if (version !== "0.0.2") {
    return new Response("version not allowed", { status: 404 });
  }

  return env.ASSETS.fetch(request);
}
```

否则后端不会关心 `v=0.0.1` 或 `v=0.0.2`。

这里的版本号只是给客户端和缓存系统看的。

## @require 的缓存按什么区分

可以把 `@require` 的缓存理解成按完整 URL 区分：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
```

脚本管理器会把这个远端依赖下载下来，并记住这个 URL 对应的 JS 内容。

所以这两个地址通常会被当成两个不同的依赖：

```txt
https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.2
```

它不是业务代码里的 `localStorage`：

```js
localStorage.setItem("key", "value");
```

也不是你能在页面 JS 里直接 `localStorage.getItem(...)` 读出来的缓存。

更准确地说，这里可能涉及几层缓存：

```txt
1. 脚本管理器自己的 @require 依赖缓存
2. 浏览器 HTTP 缓存
3. Cloudflare 边缘缓存
```

对 userscript 更新最关键的是第 1 层。脚本管理器会保存 `@require` 下载下来的依赖，这样每次打开目标网页时，不一定都要重新从远端请求一次。

可以这样理解执行过程：

```txt
安装或更新 tiny.user.js 时：
  读取 @require URL
  下载 tiny-widget.js
  保存这份依赖

每次打开目标网页时：
  先执行已保存的 tiny-widget.js
  再执行 tiny.user.js
```

所以如果远端 `tiny-widget.js` 改了，但 `@require` 还是：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
```

脚本管理器可能继续使用已经保存过的旧依赖。

当你改成：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.2
```

它看到的是一个新的依赖 URL，通常就会重新下载。

简单记：

```txt
@require 缓存键 = @require 后面的完整 URL
```

## user.js 的 @version 和 @require 的 v

它们解决的是两个不同问题：

```txt
@version
  控制脚本管理器是否认为 tiny.user.js 本身有新版本。

@require 后面的 ?v=...
  控制远端依赖 tiny-widget.js 是否换成新的下载地址。
```

### 只改 @version 可以吗

可以，但要看你改的是哪一部分。

如果只改 `tiny.user.js` 自己的正文，例如：

```js
TinyFloatDemo();
```

或者改 userscript 里的配置、调用参数，那么只改：

```js
// @version 0.0.2
```

通常就够了。脚本管理器发现 `.user.js` 有新版本，会更新 userscript 正文。

如果改的是远端依赖：

```txt
https://notes.edmund.xin/demo/tiny-widget.js
```

但 `@require` 仍然是：

```js
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
```

那么只改：

```js
// @version 0.0.2
```

不一定可靠。脚本管理器可能更新了 `tiny.user.js`，但继续使用已经缓存过的：

```txt
tiny-widget.js?v=0.0.1
```

所以如果改了 widget，建议两个都改：

```js
// @version 0.0.2
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.2
```

规则可以这样记：

```txt
只改 user.js 逻辑：
  改 @version 就行

改了 @require 指向的 widget.js：
  改 @version，也改 @require 后面的 ?v
```

如果只改了 `tiny-widget.js`，但 `@require` 地址还是：

```txt
/demo/tiny-widget.js?v=0.0.1
```

用户本地可能继续使用缓存里的旧 widget。

所以发布新 widget 时，建议同时改：

```js
// @version 0.0.2
// @require https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.2
```

这样脚本管理器会先发现 `tiny.user.js` 有新版本，然后新的 `@require` URL 也会让远端 widget 重新下载。

## 最小代码对照

`tiny.user.js`：

```js
// ==UserScript==
// @name         Tiny Float Demo
// @version      0.0.1
// @match        http://*/*
// @match        https://*/*
// @require      https://notes.edmund.xin/demo/tiny-widget.js?v=0.0.1
// @run-at       document-idle
// @grant        none
// ==/UserScript==

TinyFloatDemo();
```

`tiny-widget.js`：

```js
function TinyFloatDemo() {
  var box = document.createElement("div");
  box.textContent = "浮";
  box.style.cssText = "position:fixed;left:16px;top:16px;width:40px;height:40px;background:red;color:#fff;z-index:999999;display:grid;place-items:center;";
  document.body.appendChild(box);
}
```

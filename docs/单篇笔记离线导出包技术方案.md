# 单篇笔记离线导出包技术方案

## 1. 背景与目标

当前笔记正文已经使用 Markdown 保存，图片和其他附件通过 R2 存储，并在 Markdown 中写入远程 URL。直接导出单个 `.md` 文件虽然实现简单，但长期使用存在两个问题：

- 离线打开时图片无法显示。
- 远程资源失效、域名变更或 Markdown 阅读器禁止远程图片时，导出的笔记只剩链接。

本方案目标是实现单篇笔记的离线导出包：

- 导出一个 zip 文件，文件名为 `笔记标题.zip`。
- zip 内包含 `笔记标题.md`。
- zip 内包含 `笔记标题.assets/` 目录，保存当前笔记实际引用到的图片和附件。
- Markdown 内的本系统附件 URL 改写为相对路径，离线解压后可以直接渲染图片。
- 主站、嵌入脚本和油猴脚本复用同一套行为。

## 2. 导出包格式

### 2.1 目录结构

导出包固定使用如下结构：

```text
笔记标题.zip
├── 笔记标题.md
└── 笔记标题.assets/
    ├── 001-image.png
    ├── 002-photo.jpg
    └── 003-file.pdf
```

Markdown 中的图片引用改写为：

```markdown
![图片](笔记标题.assets/001-image.png)
```

普通附件链接改写为：

```markdown
[文档](笔记标题.assets/003-file.pdf)
```

视频和音频如果正文中是 HTML 标签，则改写 `src`：

```html
<video controls src="笔记标题.assets/004-video.mp4"></video>
```

### 2.2 文件命名规则

导出基名来自笔记标题：

- 标题去除首尾空白。
- 文件系统不安全字符替换为短横线或删除。
- 连续空白折叠为单个空格。
- 过长标题截断到合理长度，例如 80 个字符。
- 空标题兜底为 `未命名笔记`。

示例：

| 笔记标题 | zip 文件 | md 文件 | assets 目录 |
| --- | --- | --- | --- |
| `产品方案` | `产品方案.zip` | `产品方案.md` | `产品方案.assets/` |
| `接口/鉴权:记录` | `接口-鉴权-记录.zip` | `接口-鉴权-记录.md` | `接口-鉴权-记录.assets/` |
| 空标题 | `未命名笔记.zip` | `未命名笔记.md` | `未命名笔记.assets/` |

附件文件名优先保留用户上传时的原始文件名，但需要做安全清洗。为避免重名和排序不稳定，统一增加三位序号前缀：

```text
001-image.png
002-image.png
003-report.pdf
```

## 3. 功能边界

### 3.1 本版本实现

- 只导出单篇笔记。
- 导出格式固定为 zip。
- zip 内固定包含 `笔记标题.md`。
- 当前 Markdown 中实际引用到的本系统附件会被下载并放入 `笔记标题.assets/`。
- 本系统附件 URL 会被改写为相对路径。
- 外部 URL 保留原样，不自动下载第三方资源。
- 详情页导出当前编辑器内容，列表左滑导出已保存内容。

### 3.2 本版本不实现

- 不做全部笔记批量导出。
- 不做导入恢复。
- 不做 HTML 或 PDF 导出。
- 不把图片转成 base64 内嵌到 Markdown。
- 不下载第三方站点图片。
- 不新增 Worker 专用导出接口。
- 不修改 D1、R2 表结构或 binding。

## 4. 当前工程基础

当前工程中与导出相关的核心模块：

| 模块 | 位置 | 现状 |
| --- | --- | --- |
| 笔记抽屉和详情页 | `client/shared/FloatingNotesCore.tsx` | 主站、嵌入脚本、油猴脚本共用的笔记列表和详情页 |
| 笔记类型 | `client/shared/types.ts` | `Note` 已包含 `title`、`markdown`、`assetCount` |
| 笔记 API | `client/shared/notesApi.ts` | 已有 `listNoteAssets`、`uploadNoteAsset`、`deleteNoteAsset` |
| AI 聊天 bridge | `client/app/ChatApp.tsx` | 嵌入跨域时代理笔记 list/create/update/delete/uploadAsset |
| Worker 附件内容接口 | `src/index.ts` | 已有 `/api/notes/:noteId/assets/:assetId/content` |
| Widget 入口 | `client/widget/entry.tsx` | 统一构建 Shadow DOM widget |
| 嵌入入口 | `public/embed/inject-floating-notes.js` | 动态加载版本化 widget |
| 油猴入口 | `public/floating-notes.user.js` | 通过 `@require` 加载版本化 widget |

现有附件内容接口已经能从 R2 读取对象并返回内容，同时带有 CORS 头。导出时应优先通过这个接口下载附件，而不是直接拉取 `assets.notes.edmund.xin` 的公开 URL。这样可以避免跨域下载限制，也能在公开域名变更时减少导出逻辑耦合。

## 5. 技术方案

### 5.1 前端 zip 生成

新增导出模块：

```text
client/shared/exportNote.ts
```

建议使用 `fflate@0.8.3` 在浏览器端生成 zip：

- 体积小。
- 支持浏览器环境。
- API 简单，适合当前单篇导出场景。

新增依赖后需要更新：

```text
package.json
package-lock.json
```

导出函数建议形态：

```ts
type ExportNoteBundleInput = {
	noteId: string;
	title: string;
	markdown: string;
	assets: NoteAsset[];
	apiBase: string;
	fetchAssetContent: (noteId: string, assetId: string) => Promise<Blob>;
};

type ExportNoteBundleResult = {
	fileName: string;
	blob: Blob;
};
```

`FloatingNotesCore` 负责 UI、toast、读取当前编辑器 Markdown；`exportNote.ts` 只负责命名、下载、路径重写和 zip 生成。

### 5.2 附件识别

导出时先获取当前笔记附件列表：

```ts
listNoteAssets(noteId)
```

然后只打包当前 Markdown 中实际出现的附件。匹配来源包括：

- `asset.publicUrl`
- 本地开发场景下的 `/api/notes/:noteId/assets/:assetId/content`
- 完整同源 URL 形式的 `/api/notes/:noteId/assets/:assetId/content`

不在 Markdown 中出现的历史附件不打包，避免导出无关文件。

### 5.3 附件下载

每个被引用附件通过现有内容接口下载：

```text
/api/notes/:noteId/assets/:assetId/content
```

前端拼接时遵循 `apiBase`：

```text
${apiBase}/api/notes/${noteId}/assets/${assetId}/content
```

同源主站可以直接请求。跨域 widget 需要通过 bridge 请求，见第 6 节。

下载成功后写入 zip：

```text
笔记标题.assets/001-image.png
```

### 5.4 Markdown 路径重写

路径重写规则：

- 将正文中匹配到的本系统附件 URL 替换为相对路径。
- 相对路径不做 URL 编码，保持可读性，例如 `产品方案.assets/001-image.png`。
- 如果文件名包含空格，Markdown 仍可正常解析；必要时保留原 Markdown 的括号形式。
- 外部 URL 原样保留。

示例：

导出前：

```markdown
![截图](https://assets.notes.edmund.xin/users/u1/notes/n1/a1-image.png)

![外部图](https://example.com/a.png)
```

导出后：

```markdown
![截图](产品方案.assets/001-image.png)

![外部图](https://example.com/a.png)
```

### 5.5 下载触发

生成 zip 后使用浏览器 Blob 下载：

```ts
const url = URL.createObjectURL(zipBlob);
const anchor = document.createElement("a");
anchor.href = url;
anchor.download = "笔记标题.zip";
anchor.click();
URL.revokeObjectURL(url);
```

导出成功显示 toast：

```text
已导出
```

失败显示具体错误：

```text
导出失败
附件下载失败
附件上传中，请稍后导出
```

## 6. 主站、嵌入和油猴兼容

### 6.1 主站

主站中 `FloatingNotesCore` 与 Worker API 同源，可以直接调用：

- `listNoteAssets(noteId, apiBase)`
- `fetchWithAuth(apiBase, /api/notes/:noteId/assets/:assetId/content)`

### 6.2 嵌入和油猴

第三方页面中的 widget 可能与 `notes.edmund.xin` 跨域。当前工程已经通过 AI 聊天 iframe 中的 `ChatApp` 代理笔记操作。

导出需要补充 bridge action：

```ts
type NoteBridgeRequest =
	| { action: "list" }
	| { action: "create"; payload: { note: DraftNote } }
	| { action: "update"; payload: { id: string; note: DraftNote } }
	| { action: "delete"; payload: { id: string } }
	| { action: "uploadAsset"; payload: { id: string; file: File } }
	| { action: "listAssets"; payload: { id: string } }
	| { action: "downloadAsset"; payload: { noteId: string; assetId: string } };
```

`listAssets` 返回 `NoteAsset[]`。

`downloadAsset` 建议返回 `ArrayBuffer` 或 `Blob`。如果 postMessage 对 Blob 兼容性存在问题，可返回：

```ts
{
	fileName: string;
	mimeType: string;
	buffer: ArrayBuffer;
}
```

前端再把 `ArrayBuffer` 转成 `Blob` 写入 zip。

### 6.3 版本同步

实现代码时需要同步 bump widget 版本。当前版本是 `1.0.22`，建议下一版为 `1.0.23`：

- `client/widget/entry.tsx`
- `public/embed/inject-floating-notes.js`
- `public/floating-notes.user.js`
- userscript `@require` 的 `v=1.0.23`

构建后需要确认：

```text
public/embed/floating-notes-widget.js
```

也包含新版本逻辑。

## 7. UI 放置

### 7.1 详情页头部

详情页头部增加一个 `FileDown` 图标按钮，放在标题输入框和保存按钮附近。

建议顺序：

```text
返回 | 标题输入 | 导出 | 保存
```

原因：

- 导出属于当前笔记详情操作，语义明确。
- 可以导出尚未保存但已经在编辑器中的内容。
- 不需要用户回到列表才能导出。

### 7.2 列表左滑

列表左滑操作增加“导出”：

```text
问AI | 复制 | 导出 | 删除
```

现有 `.swipe-item.open .note-item` 位移需要从三个按钮宽度调整为四个按钮宽度：

```css
.swipe-item.open .note-item {
	transform: translateX(-288px);
}
```

按钮宽度沿用当前 `72px`。

## 8. 异常处理

| 场景 | 处理 |
| --- | --- |
| 当前是新笔记且没有 `noteId` | 如果无附件，直接导出 zip；如果有待上传附件，提示先等待上传完成 |
| Markdown 中包含 `blob:` 或上传占位 | 等待正在进行的上传；仍存在占位则阻止导出 |
| 当前笔记没有附件 | 导出 zip，里面只有 `笔记标题.md` |
| 附件列表有文件但正文未引用 | 不打包 |
| 某个引用附件下载失败 | 阻止生成 zip，提示“附件下载失败” |
| 外部图片 URL | 保留原 URL，不下载 |
| 文件名重名 | 使用序号前缀保证唯一 |
| 标题为空 | 使用 `未命名笔记` |
| 用户未登录 | 沿用现有笔记加载错误提示 |

## 10. Cloudflare 边界

本方案不新增 Worker 导出接口，主要在浏览器端完成 zip 生成，因此不会引入新的 Worker CPU、内存或响应体压力。

仍需注意：

- 附件内容通过现有 Worker 内容接口读取，每个附件会产生一次请求。
- 大附件会占用浏览器内存，第一版不做超大文件流式 zip。
- Worker bindings 不变，因此不需要执行 `npx wrangler types`。
- D1/R2 schema 不变，因此不需要新增 migration。

本方案参考的 Cloudflare 文档：

- Workers limits: https://developers.cloudflare.com/workers/platform/limits/
- Workers Response API: https://developers.cloudflare.com/workers/runtime-apis/response/
- Workers Static Assets routing: https://developers.cloudflare.com/workers/static-assets/routing/

## 11. 后续扩展

后续可以在不破坏当前导出格式的基础上扩展：

- 全部笔记批量导出为一个总 zip。
- 增加 `manifest.json` 保存笔记 id、创建时间、更新时间、附件元数据。
- 支持导入 zip 恢复笔记和附件。
- 支持导出 HTML。
- 支持浏览器打印成 PDF。
- 支持仅导出 Markdown，不带附件，用于快速分享。
